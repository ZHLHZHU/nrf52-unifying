# 优联发射功能迁移记 —— 从树莓派+nRF24 到 nRF52840 原生射频

> 这是一篇偏分享性质的文档,记录把 Logitech Unifying 键盘发射功能从"树莓派 + nRF24L01 模块"迁移到"nRF52840 片上原生射频"的探索过程,包括踩过的坑和最终怎么走通的。

## 背景

手头有两个东西:

- `rust-unifying`:一个已经把 Unifying 协议(配对、加密键击、保活、HID++)用 Rust 重写过的项目。它原本跑在树莓派上,通过 SPI 驱动外接的 nRF24L01 模块发射。好在它的设计很干净——协议逻辑完全硬件无关,藏在三个 trait 后面:`UnifyingRadio`(射频)、`Clock`(毫秒时钟)、`AesEncryptor`(AES-128)。
- `nrf52-rs-scaffold`:一个 nRF52840 的 embassy 脚手架,已经跑通了 USB-CDC OTA(首刷之后就能纯靠串口升级固件,不用 SWD)。

目标:让 nRF52840 用**自己的片上 2.4GHz 射频**直接当优联键盘,不要外接 nRF24 模块,所有操作走 CDC 串口。

一个很关键的有利条件:nRF52840 通过 CDC 连在同一台 ARM Linux 主机上,而 Logitech 接收机(046d:c52b)也插在这台机器上。这意味着我可以在同一台机器上完成"发固件→发命令→看接收机收到什么"的完整闭环,验证起来非常快。

## 核心洞察:Unifying 跑的就是 ESB

第一个让整件事变得可行的认识是:**Logitech Unifying 的空中协议本质就是 Nordic 的 Enhanced ShockBurst(ESB)**,也就是 nRF24L01+ 用的那套。而 nRF52840 的片上 RADIO 外设可以配置成与 nRF24L01+ 字节级兼容的 ESB。

所以迁移的真正工作量,**只有一件事**:在 nRF52840 的原生 RADIO 上重新实现 `UnifyingRadio` 这一个 trait。协议层、加密、配对状态机,统统原样复用。

剩下的 `Clock` 用 `embassy-time`、`AesEncryptor` 用 `aes`+`ctr` crate,都是十几行的胶水代码。

## 第一个坑:工具链

刚开始连基线都编不过。`fixed@1.31.0` 这个传递依赖要求 rustc 1.93,而手头是 1.89。处理办法很简单——`cargo update fixed --precise 1.29.0` 锁到一个兼容版本。

还有 `rust-unifying` 库本身:它默认依赖 `rppal`(树莓派 GPIO)和 `rf24-rs`,这些都进不了 `no_std` 的嵌入式环境。把这些依赖全部 gate 到 `rf24` feature 后面,库就能干净地以 `no_std` 编进 nRF52 固件了。

## 啃硬骨头:手写 ESB PTX 驱动

`esb` crate(2020 年的)倒是有现成的 nRF52 ESB 实现,但它绑死了一个很老的 PAC 版本,跟 embassy-nrf 用的 PAC 冲突,没法直接用。

所以我决定直接对着 embassy 自带的 PAC 手写一个最小的 ESB PTX(主发射端)驱动。好在那个老 `esb` crate 的**寄存器配置**仍然是极好的参考——它是经过实测能跟真实 nRF24L01+ 互通的。从它那里抄到了关键配置:

- 2Mbit Nordic 私有模式
- 包格式:6-bit 长度字段 + 3-bit S1(PID + no_ack),无 S0
- CRC-16(多项式 0x11021,初值 0xFFFF)
- 大端、关闭白化
- **地址要做位反转**(nRF24L01+ 在硬件里做这个,nRF52 得在软件里补)

我的驱动做成了阻塞式的"发一包 → 转 RX 收 ACK → 没收到就重传",因为 CDC 命令模型本来就是同步的。一开始 TX→RX 转向我用了硬件 shortcut 自动跳转,但那有个竞态(要在 ACK 到达前重新指 DMA 缓冲)。后来改成全序列化:发完等 DISABLED,再手动开 RX,反正接收机回 ACK 要等 ~130μs,而快速 ramp-up 只要 ~40μs,时间余量很充足。

## 最难定位的坑:地址编码

驱动写完,第一次发 `UPAIR`——**零 ACK**。射频在发(信道号一路从 5 跳到 74),但接收机一个都不回。

这是手写 ESB 最容易踩、也最难调的地方:**地址的字节序和位序**。nRF52 的 BASE+PREFIX 拆分方式、字节顺序、位序,跟 nRF24L01+ 的 TX_ADDR 寄存器布局都不一样,有好几种看似合理的组合。

如果靠"改一种猜测 → 重新编译 → OTA 刷入 → 测一次"来试错,每轮要 3 分钟,太慢了。

所以我换了个思路:**在一份固件里同时内置 4 种候选地址编码**,加一条 `URADIOTEST` 诊断命令,让它在配对地址上对每种编码扫频统计 ACK 数。一次刷入,跑一遍:

```
RADIOTEST ENC=0 ACKS=0
RADIOTEST ENC=1 ACKS=2   ← 就是它
RADIOTEST ENC=2 ACKS=0
RADIOTEST ENC=3 ACKS=0
```

编码 1 拿到了 ACK,其余全 0。一次性经验性地定位了正确编码,完全不用靠猜。这个"把不确定性塞进一份固件、让硬件来投票"的办法,后来回想起来是整个迁移里最划算的一步。

把编码 1 设为默认,重刷,`UPAIR` 立刻成功:

```
UPAIR -> PAIRED 3D41950E0C CH=32
```

接收机的内核日志也出现了 `Found new device, id=0x03 Keyboard`。

## 怎么确认"真的能用",而不只是"ACK 了"

ESB 收到 ACK 只能证明射频帧被接收机收到了,**不能证明接收机成功解密并把按键注入了系统**。这两件事差很远。

我的验证办法是用 Linux 的 `EVIOCGRAB` **独占抓取**接收机那个键盘 event 节点(这样按键不会泄漏到终端),同时发 `UTYPE hello`,然后看抓到了什么:

```
UTYPE hello -> TYPED 5/5
CAPTURED: hello
```

`CAPTURED: hello` 是铁证——这五个字符确确实实是从无线接收机解密出来、注入到系统的。从原生射频 → 配对 → AES 密钥派生 → 加密键击 → 接收机解密 → HID 注入,整条链路全通了。

## 持久化:AES counter 是安全关键

配对状态本来只在 RAM 里,一复位就没了,每次 OTA 后都得重新配对,很烦。于是把配对信息(地址、AES 密钥、信道、counter)存进 STORAGE 分区。

这里有个不能想当然的点:**AES counter 必须持久化,而且复位后要续用更大的值**。它是接收机的防重放计数,如果复位后从 0 重来,接收机会因为"这个 counter 我见过"而拒收键击。所以每次输入后都要把递增的 counter 存回去。

为了不磨损 flash,存储用的是页内追加式日志:一页 4K 分成 128 个 32 字节槽位,每次存追加一条,写满才擦页。加载时扫到最后一条有效记录。

验证也是端到端的:配对→输入(CNT=10)→**重启**→不重新配对→直接连接输入,抓到 `CAPTURED: world`,counter 从 10 续到 20。重启后配对完好、密钥可用、防重放正确。

## 最深的坑:陈旧链路丢键(以及我一开始修错了)

加上鼠标(后来按需求砍掉了)和组合键之后,发现一个间歇性问题:命令之间隔几秒再发,经常返回 `ERR SEND`,按键丢失。

我的第一反应是"链路变陈旧了,失败后自愈一下"——发送失败就调 `connect()` 重新扫频再重试。改了。结果**更糟**:多轮重试场景下几乎全失败。

这时候我停下来了。按经验,同一个思路连改两次还更差,就不该继续盲打补丁,而要回去找根因。

重新梳理所有证据后,真正的根因浮出水面:**优联连接是靠持续不断的 keep-alive 包维持的**。真实键盘每 ~20ms 发一个保活包,接收机据此维持跳频同步。一旦几秒没保活,接收机直接判定设备掉线。

而我的架构是"每条 CDC 命令:开串口→发一次→关",**命令之间设备停在 `read_packet().await` 上,什么都不发**。所以紧跟 connect 的命令成功(链路新鲜),隔几秒的命令必败(接收机已判掉线)。而"事后 connect 补救"根本不可靠——接收机处于"已掉线"状态时的响应窗口跟配对模式不一样,单发 wake-up 经常追不上,多轮重试还会互相打断接收机正在重建的同步,所以越改越糟。

**正确的修法是从源头维持连接,而不是事后补救。** 用 embassy 的惯用模式 `select(read_packet, keepalive_timer)`:在等待下一条命令的同时,每 8ms 穿插一次 `tick()`。`tick()` 内部自己判断保活窗口是否到期,大多数是空操作,到期才真正发包——正好复刻真实键盘的保活节奏,而且单任务单所有者,没有跨 await 的借用问题。

改完用同样的"6 秒间隔"场景验证:

```
gap6s key#1..6 f5 -> OK (6/6)
CAPTURED: F5F5F5F5F5F5
```

6/6 全中。彻底解决。

## 回头看

几个比较有价值的体会:

- **分层干净的代码迁移起来事半功倍。** `rust-unifying` 把硬件藏在三个 trait 后面,所以这次迁移真正的工作量只压在一个 `UnifyingRadio` 实现上。
- **不确定性别靠猜+重刷,塞进一份固件让硬件投票。** 地址编码那一关,`URADIOTEST` 一次性解决了本来要试错很多轮的问题。
- **"ACK 了"不等于"能用"。** 用 EVIOCGRAB 独占抓取做端到端验证,才能确认解密注入真的发生了,而不是停在射频层自我感觉良好。
- **同一思路失败两次就停下找根因。** 陈旧链路那个坑,如果继续在"事后自愈"上打补丁会越陷越深;退回去看清楚是"缺乏持续保活"这个架构问题,才一次修对。

最终设备就是个稳定、不丢键、配对持久化、纯片上射频的无线键盘,所有操作走一根 USB 线。
